## 费曼技巧：用一句话解释

**s09 - Agent Teams**：持久身份（config.json 花名册）+ JSONL 邮箱（每人一个收件箱），让多个 Agent 并行工作、异步通信，队友不再用完即扔。

**s10 - Team Protocols**：自由通信缺少协商规则，引入 FSM（pending→approved/rejected）和 request_id 关联机制，让关机和高风险操作有结构化握手协议。

**s11 - Autonomous Agents**：队友变 idle 后主动轮询任务图，自动认领无前置依赖的未分配任务——从领导指派变成自组织，通信从推模式变为拉模式，长期运行后压缩触发需身份重注入。

**s12 - Worktree Isolation**：每个任务分配独立 git worktree 目录，控制平面（.tasks/）管"做什么"，执行平面（.worktrees/）管"在哪做"，事件流记录生命周期时间线，崩溃后可重建现场。

## 第四阶段核心思维漏洞

### 混淆"协商协议"和"任务进度"

**出现位置**：s10 开始时描述"解决了不知道任务状态执行到哪"——这是 s07 的问题。

**正确区分**：
- s07：任务执行进度持久化（做到哪了）
- s10：Agent 间协商结构化（怎么谈判）

**规律**：每次遇到新机制时，要先问"这解决的是状态管理问题还是协调问题"，两者容易混淆。

### 推/拉模式的直觉

**出现位置**：s11 身份重注入的原因，以及推/拉选择的根本逻辑

**正确模型**：
- 推模式：有天然发送方（后台线程知道自己完成了）
- 拉模式：无天然发送方（没有人知道"有新任务可认领"）→ 接收方自己定时去问

## 全课程架构演进图

```
s01: while True + stop_reason          ← 最小可行 Agent（30行）
      ↓
s02: + dispatch map                    ← 开闭原则：工具扩展不改核心
      ↓
s03: + TodoManager + nag reminder      ← 对抗注意力稀释（内存）
      ↓
s04: + run_subagent() + 独立messages   ← 上下文隔离
      ↓
s05: + SkillLoader + YAML frontmatter  ← 两层知识架构
      ↓
s06: + micro/auto/manual compact       ← 三层压缩策略
      ↓
s07: + TaskManager + DAG 依赖图        ← 跨会话持久化（磁盘）
      ↓
s08: + BackgroundManager + 通知队列    ← 异步执行（不阻塞主循环）
      ↓
s09: + TeammateManager + MessageBus    ← 持久身份 + JSONL 邮箱
      ↓
s10: + FSM + request_id                ← 结构化协商协议
      ↓
s11: + idle_poll + 身份重注入          ← 自组织 Agent
      ↓
s12: + WorktreeManager + events.jsonl  ← 控制平面/执行平面分离
```

**状态外化路径**：
```
内存 messages → 磁盘 .tasks/ → 磁盘 .worktrees/ → git history
越往后：状态越持久，越能在崩溃后重建
```

## Harness 工程通用原则（全课程提炼）

| # | 原则 | 体现的课程 |
|---|------|-----------|
| 1 | **开闭原则**：往 dispatch map 加条目，核心循环永远不变 | s02，贯穿全课 |
| 2 | **状态外化**：会话/内存是易失的，磁盘是可靠的 | s07, s12 |
| 3 | **Harness 保证不变量，LLM 做决策**：注入信息，不替 LLM 思考 | s03, s05 |
| 4 | **分层管理不同粒度**：知识两层（s05）、压缩三层（s06）、状态快照+时间线（s12） | s05, s06, s12 |
| 5 | **不阻塞主流程，完成后通知**：推模式（s08）或轮询（s11） | s08, s11 |
| 6 | **控制平面/执行平面分离**：管"做什么"和管"在哪做"解耦 | s12，类比 K8s |
| 7 | **原子操作减少中间状态**：多步合并为一，最小化不一致窗口 | s12 |
| 8 | **物理隔离优于逻辑锁**：独立目录比加锁更简单可靠 | s04, s12 |
| 9 | **请求-响应用关联 ID**：多并发请求时，响应必须能找到对应请求 | s10，类比 HTTP/RPC |
| 10 | **极端操作往往错，正确答案在分层中间**：选择性清理，不全清不全留 | s06 |

## 三个维度的突破

这 12 节课让 LLM 突破了三个维度的限制：

**时间维度**（任务比一次对话更长）：
- s07：任务持久化，跨会话续跑
- s08：异步执行，不阻塞等待

**空间维度**（任务比一个上下文窗口更大）：
- s04：子 Agent 隔离上下文
- s05：知识按需加载
- s06：压缩释放空间

**规模维度**（任务比一个 Agent 更复杂）：
- s09：多 Agent 通信
- s10：结构化协商
- s11：自组织认领
- s12：文件系统隔离

## 如果重新设计，我会……

1. **从 s07 就考虑并发**：s07 的任务图没有锁，s11 靠运气，s12 靠文件系统唯一性——如果重新设计，会从一开始就把并发安全设计进去

2. **事件流从 s09 就引入**：多 Agent 系统一旦出现，调试就变难了，事件流应该是基础设施，不是 s12 才加的

3. **身份管理统一**：s09 的 config.json、s11 的身份重注入、s10 的 request_id 都是"身份"的不同侧面，统一设计会更清晰
