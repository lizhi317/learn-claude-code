## 费曼技巧：用一句话解释

**s07 - Task System**：把任务从内存扁平清单升级为磁盘上的 DAG，通过 `blockedBy` 依赖关系让 Agent 知道什么能做、什么被卡住，目标跨越会话边界存活。

**s08 - Background Tasks**：后台线程执行慢 I/O，完成后推入通知队列，主循环每次 LLM 调用前排空队列——LLM 不阻塞，I/O 并行化。

## 核心思维漏洞复盘

### 异步直觉不够准

**出现位置**：s08 超时那题——认为"超时会让主 Agent 一直在等"

**正确模型**：后台线程本来就是异步的，主循环不等它。超时的真正后果是：
- 后台线程变僵尸线程，资源泄漏
- 通知队列永远没有该任务的结果
- 依赖该任务的后续任务永远被阻塞

**根因**：遇到"任务卡死"，第一反应是"调用方在等"（同步思维），而不是"资源泄漏 + 队列永远空"（异步思维）。

---

## 两机制协同：为第四阶段奠基

s07 和 s08 联合回答了一个问题：**如何让 Agent 高效处理比一次对话更大、更长的工作？**

```
大任务
  │
  ├─ [s07 TaskManager] 拆成 DAG，持久化到磁盘
  │   └─ 跨会话续跑：status + blockedBy 告诉 Agent 从哪继续
  │
  └─ [s08 BackgroundManager] 慢操作扔后台
      └─ 主循环不阻塞：drain_notifications 在 LLM 调用前注入结果
```

更重要的是，这两个机制共同为第四阶段（多 Agent 团队）铺了路：

| s07/s08 埋下的伏笔 | 第四阶段真正用上的地方 |
|---------------------|------------------------|
| `owner` 字段（任务归属） | s09 多 Agent 认领任务，防止重复执行 |
| 任务图共享状态 | s09 多 Agent 读写同一个任务图协调工作 |
| 异步完成通知 | s09 Agent 间通信（MessageBus）的设计基础 |

## 第三阶段可泛化原则

| 原则 | 体现 |
|------|------|
| 状态外化到可靠存储 | 任何跨进程/跨会话的系统都需要把状态持久化（数据库、文件、KV） |
| DAG 调度 | CI/CD、数据管道、构建系统都用同一个模式：有向无环图 + 依赖解除 |
| 不阻塞主流程，完成后通知 | Kafka、async/await、I/O 中断——异步系统的通用模式 |
| 协调中心模式 | 多执行者共享一份状态存储，靠字段（status/owner）协调，不需要点对点通信 |
